## Critical Findings

### C1: `/api/auth/register` â€” No Authentication Required
**Severity:** CRITICAL
**File:** `src/app/api/auth/register/route.ts`
**Issue:** Endpoint triggers Neynar FID registration with zero authentication. Anyone who knows a wallet address on the allowlist can call this route. No Zod validation on inputs. No session created after registration.
**Fix:** Add session check or require verified wallet signature before processing.

## High Findings

### H1: In-Memory Nonce Stores (Serverless Incompatible)
**Severity:** HIGH
**Files:** `src/app/api/auth/verify/route.ts`, `src/app/api/auth/siwe/route.ts`
**Issue:** Both auth routes use in-memory `Map()` for nonce storage. In Vercel serverless (multiple isolates), a nonce generated by isolate A won't exist in isolate B. Legitimate logins fail when GET and POST hit different isolates.
**Fix:** Move nonce storage to Redis, Supabase, or Vercel KV.
**Note:** This is a reliability issue, not a replay vulnerability â€” replayed nonces are rejected because they don't exist in any isolate's store.

### H2: `chat/schedule` POST â€” No Zod Validation
**Severity:** HIGH
**File:** `src/app/api/chat/schedule/route.ts`
**Issue:** POST handler destructures body directly without Zod. `embedHash`, `embedUrls`, `crossPostChannels`, and `scheduledFor` are taken raw. If `crossPostChannels` is not an array, `.filter()` will throw.
**Fix:** Add Zod schema matching `sendMessageSchema` plus `scheduledFor`.

### H3: `auth/register` â€” No Zod, No Session Creation
**Severity:** HIGH
**File:** `src/app/api/auth/register/route.ts`
**Issue:** `walletAddress`, `signature`, `deadline`, `fname` taken raw from body. No session created after successful registration.
**Fix:** Add Zod schema. Create session after registration.

### H4: No Centralized Auth Guard in Middleware
**Severity:** HIGH
**File:** `src/middleware.ts`
**Issue:** Middleware only does rate limiting and headers. Every API route must manually add session checks. A new route without `getSessionData()` at the top is publicly accessible by default.
**Fix:** Add middleware-level auth check for `/api/` routes (with explicit allowlist for public endpoints like webhooks, register, session).

## Medium Findings

### M1: Wildcard Injection in Chat Search
**File:** `src/app/api/chat/search/route.ts:89`
**Issue:** User search query passed to `.ilike('text', '%${query}%')` without escaping `%` and `_`. Searching `%` returns all messages.
**Fix:** Escape `%` â†’ `\%` and `_` â†’ `\_` before interpolation.

### M2: Neynar Error Messages Proxied to Client
**File:** `src/app/api/chat/react/route.ts:49,116`
**Issue:** Upstream Neynar error messages forwarded directly. Could leak internal API details.
**Fix:** Return generic "Reaction failed" message.

### M3: No Rate Limit on `/api/music/metadata`
**File:** `src/middleware.ts`
**Issue:** Endpoint makes outbound HTTP calls to 9 third-party APIs. No rate limit = potential abuse.
**Fix:** Add `'/api/music/metadata': { limit: 20, window: 60 }` to middleware.

### M4: `/api/music/radio` â€” No Auth, No Rate Limit
**File:** `src/app/api/music/radio/route.ts`
**Issue:** Only endpoint with no `getSessionData()` check. Publicly accessible. No rate limit.
**Fix:** Add session check. Add rate limit.

### M5: Miniapp Webhook FID Spoofing
**File:** `src/app/api/miniapp/webhook/route.ts`
**Issue:** No signature verification (Farcaster Mini App spec doesn't support HMAC). Anyone can register/disable notification tokens for any allowlisted FID.
**Fix:** Platform limitation. Add rate limiting and consider additional verification.

### M6: Members Endpoint Exposes All Addresses
**File:** `src/app/api/members/route.ts:103-108`
**Issue:** Returns `wallet_address`, `custody_address`, `verified_addresses`, `xmtp_address`, and `lastLoginAt` for ALL members to any authenticated user.
**Fix:** Consider limiting to XMTP-reachable addresses only, or removing `lastLoginAt`.

### M7: Governance Category Enum Mismatch
**File:** `src/app/(auth)/governance/page.tsx` vs Zod schema
**Issue:** Frontend sends "music"/"tech" but Zod schema expects "technical"/"community". Proposals with those categories silently fail.
**Fix:** Align frontend category values with Zod schema.

### M8: Zero-Weight Votes Allowed
**File:** `src/app/api/proposals/vote/route.ts`
**Issue:** Users with 0 Respect tokens can vote (weight = 0). Inflates voter count without affecting outcome.
**Fix:** Decide: block zero-weight votes, or allow them as non-binding participation.

### M9: `ogPct`/`zorPct` Show `undefined%`
**File:** `src/app/(auth)/governance/page.tsx:214,219`
**Issue:** API strips these fields but UI renders them as `{myEntry.ogPct}%`.
**Fix:** Either include fields in API response or remove from UI.

### M10: No Zod on Auth Verify/SIWE Bodies
**Files:** `src/app/api/auth/verify/route.ts:43`, `src/app/api/auth/siwe/route.ts:44`
**Issue:** Request bodies destructured without Zod validation.
**Fix:** Add Zod schemas for both routes.

### M11: No Zod on Notifications PATCH
**File:** `src/app/api/notifications/route.ts:64-77`
**Issue:** `body.ids` used in `.in('id', body.ids)` without type/format validation.
**Fix:** Add `z.object({ ids: z.array(z.string().uuid()) }).or(z.object({ all: z.literal(true) }))`.

### M12: Register Wallet Not Normalized
**File:** `src/app/api/auth/register/route.ts:14`
**Issue:** `walletAddress` not lowercased before allowlist check. Case-sensitive comparison could miss matches.
**Fix:** `.toLowerCase()` before comparison.

### M13: Signer FID Check Bypassed When Falsy
**Files:** `src/app/api/auth/signer/status/route.ts:21`, `src/app/api/auth/signer/save/route.ts:31`
**Issue:** `if (status.fid && status.fid !== sessionData.fid)` â€” skipped when `status.fid` is undefined (pending signer).
**Fix:** Change to `if (status.fid !== undefined && status.fid !== sessionData.fid)`.

### M14: TOCTOU Race on Proposal Vote
**File:** `src/app/api/proposals/vote/route.ts:49-101`
**Issue:** Proposal status check and vote insert are not atomic. A vote could land on a just-closed proposal.
**Fix:** Low risk for 40-member community. Could use Supabase RPC for atomic check+insert.

### M15: Comments Allowed on Closed Proposals
**File:** `src/app/api/proposals/comment/route.ts:69-81`
**Issue:** No check that proposal is open before accepting a comment.
**Fix:** Add status check before insert.

## What's Secure (Confirmed)

- iron-session cookies: httpOnly, secure, sameSite=lax, 7-day TTL
- XMTP keys never leave the browser (confirmed in code + comments)
- Admin routes all check `isAdmin` properly
- Notifications scoped to `session.fid` â€” no cross-user access
- Neynar webhook HMAC verification present (minor timing issue)
- Vote weight computed server-side from on-chain data â€” cannot be client-spoofed
- Respect leaderboard properly scoped
- Allowlist gating enforced on all auth flows
