FIP Live Activity

A weekend explorable explainer for what @rish dropped on 2026-05-03 and how ZAO plugs in. Click around. Nothing breaks.

1. What problem does this solve?

Right now, if you're in an audio space on Farcaster (say, Juke), only people inside Juke can see you're live. Warpcast doesn't know. Coinbase Wallet doesn't know. ZAO doesn't know. Each app has to build its own presence integration.

FIP #268 fixes this by adding one slot per user on the Farcaster protocol that says "I'm currently at this URL right now." Every Farcaster client reads it. One protocol primitive, every client renders it.

Think of it like a Slack status. But cross-app. And on a decentralized protocol.

2. The heartbeat — keeping the slot fresh

The slot is just a URL string. Empty string means "not live." But a server-stored URL is forever — how does the network know you've LEFT the space and aren't there anymore?

Answer: heartbeat. Your app re-writes the same URL every 1 second. Each rewrite advances the message timestamp. If the timestamp is older than 5 seconds, readers treat the slot as "stale" — even if the URL is still there.

It's like leaving a "Be Right Back" sticky note. The note doesn't go away — but if you don't touch it every second, everyone assumes you actually left.

3. Why "5 seconds"? The freshness window

Two competing pressures:

  • Faster freshness = better UX. If you leave a space, your friends see your green dot disappear in 1 sec instead of 30 sec.
  • Slower freshness = less network load. 1-second heartbeats × millions of users = Snapchain melts.

FIP #268 picked 1-second writer + 5-second reader. That gives you 4 seconds of redundancy: a writer who misses 4 heartbeats in a row still appears live. Five misses = stale.

The math: why this is borderline

If 5,000 users go live concurrently (Discord Stage hits this at peak), that's 5,000 writes/sec just for LIVE_AT. Snapchain's current sustained throughput target is ~1,000 writes/sec. 5x over.

This is in our FIP feedback comment. ZAO is the only voice raising it.

4. The stalker problem

Here's the dark side. The slot is public by design — anyone can subscribe and see where you are right now. That's the whole point: cross-client discoverability.

But it also means an adversary can subscribe to your slot and follow you in real time. Worse: if a small space publishes its fc:live:speakerFids meta tag, an adversary cross-references "Alice is at URL X" with "URL X has speakers [Alice, Bob, Carol]" and now knows Alice is hanging out with Bob and Carol.

YOU vs. THE STALKER
Three things ZAO can do at the app layer
  • Opaque room IDs. Use /spaces/abc-uuid not /spaces/zounz-private-therapy-group. URL leaks no category info.
  • Gate speakerFids for private rooms. Only serve speaker list to authenticated participants, not in public meta tags.
  • Defer recording URL. Don't publish fc:live:recordingUrl during the live window — even pointing at an access-checked page leaks "this was recorded."

Plus: Quilibrium's Oblivious Transfer can mitigate the read side (see section 7).

5. The cooperative writer dance

Multiple apps can write to your LIVE_AT slot. Juke. ZAO. Maybe a future "presence-only" app like Recaster. They'd clobber each other every second if they all wrote unconditionally.

FIP #268 specifies a politeness convention:

  1. Read the current slot before writing
  2. If empty or stale (>5s old): write your URL
  3. If fresh and you wrote it: keep heartbeating
  4. If fresh and someone ELSE wrote it:
    • Activity URL (one with fc:live=1 meta tag) MAY overwrite a presence-only URL
    • Presence-only URL SHOULD yield to a fresh non-self URL

Try: ZAO writes presence first → Juke writes activity (overwrites). Or: Juke first → ZAO yields. The protocol enforces nothing. It's pure social pressure + the freshness rule.

6. JFS — proving you're really the host

Anyone can put <meta property="fc:live:hostFid" content="123"> on their webpage. So how does a Farcaster client trust that the URL is really hosted by FID 123?

Answer: JSON Farcaster Signature (JFS). A small signed envelope at a manifest endpoint linked from the URL. The host signs "I claim this URL" with their app key. Any client can verify it.

┌─────────────────────────────────────────────────────────┐ │ JFS ENVELOPE (3 base64url-encoded fields) │ └─────────────────────────────────────────────────────────┘ header = base64url({ fid, type:"app_key", key }) payload = base64url({ url:"https://zaoos.com/spaces/abc" }) signature = ed25519.sign("${header}.${payload}", privKey) ┌─────────────────────────────────────────────────────────┐ │ VERIFICATION (5 steps) │ └─────────────────────────────────────────────────────────┘ 1. base64url-decode header + parse JSON 2. base64url-decode payload + parse JSON 3. base64url-decode signature → 64 bytes 4. ed25519.verify(sig, "${header}.${payload}", header.key) 5. Snapchain query: is header.key in active key set for header.fid? ✓ All pass → trusted live activity by FID ✗ Any fail → render as plain link, no host claim

FIP #269 reuses the same envelope shape that Mini Apps already use for accountAssociation. Devs already know it. Code is open at farcasterxyz/miniapps/packages/miniapp-node/src/jfs.ts — ZAO lifts ~100 lines.

Spec inconsistency we caught

FIP #269 spec text says verify header.type === "auth". The actual reference implementation in jfs.ts:91 uses literal 'app_key'. One of the two has to change. Flagged in our FIP feedback comment.

7. Quilibrium Oblivious Transfer — the privacy magic trick

Cassie Heart's Quilibrium has a primitive called Oblivious Transfer that lets you read data from a server without the server learning what you read.

Imagine you walk into a library. You want book #3. Normally you ask the librarian "give me book #3" and they hand it to you — they know you read book #3.

With OT: you whisper a randomized request, they hand you 10 books, and you can only open book #3. They never learn which one you opened.

WITHOUT OT (current FIP #268) Stalker → Snapchain: "give me LIVE_AT for FID 188" Snapchain → Stalker: "https://zaoos.com/spaces/abc" Stalker now knows: FID 188 is in space abc, in real time WITH OT (Quilibrium-bridged read) Stalker → OT layer: obfuscated query (random seed) OT layer → Stalker: encrypted blob Stalker decrypts: "https://zaoos.com/spaces/abc" Network never learned: "who queried for whom"

This is production today. Quilibrium shipped Ferret-based OT in Q1 2025, 200x faster than basic OT.

Cost: ~50-200ms extra latency per query. Acceptable for "render presence in feed." Maybe too slow for instant green-dot UI.

ZAO's strategic angle: install Hypersnap, then add an OT bridge for ZAO-gated room reads. Nobody else in the FIP convo can credibly do this. Cassie engages via code, so the play is: Hypersnap PR review + Tantivy docs + cast about the bridge → tag her.
