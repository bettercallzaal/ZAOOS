## 1. Tidal — Immersive Dynamic Player (March 2026)

**What they did:** Tidal shipped a redesigned iOS full-screen player where the background dynamically shifts color to match the currently playing album art. Artwork is displayed large and square (no rounded corners), dominating the screen. Controls sit below in the same palette. The cramped bottom miniplayer was replaced entirely. Android is next. Designed by Aziz Firat (Tidal design team).

**Pillar:** Culture — listening as an aesthetic, emotional experience rather than a utility transaction.

**ZAO adaptation:** The existing `PlayerProvider.tsx` + `HTMLAudioProvider.tsx` already manages player state and MediaSession. Add a `useDominantColor` hook (use `node-vibrant` or a Canvas-based extractor) that samples the current track's artwork and pipes a CSS variable into the player shell. The full-screen player in `src/components/music/` gets a `background: linear-gradient(var(--album-primary), var(--album-dark))` layer behind controls. Cost: ~1 hook + CSS variable threading. High visual impact, zero new infra.

**Related research:** `research/289-zoe-dashboard-chat-ux-patterns/` covers ambient visual design in chat contexts.

## 2. Lens/Hey — Token-Gated Feeds via Modular Rules (April 2025)

**What they did:** Lens V3 launched on mainnet with social primitives as modular smart contracts: Feeds, Graphs, Groups, Rules, and Actions. The killer UX pattern: a creator attaches a "Rule" to their Feed requiring a micro-subscription token; fans pay once (gasless via GHO stablecoin on Lens Chain) and gain permanent feed access. Grove (Lens's storage layer) lets users edit/delete their own posts post-publication — mutability as a deliberate design choice, with on-chain access control enforcing who can modify what.

**Pillar:** Revenue + Distribution — subscription-gated content with zero-commission on-chain payments.

**ZAO adaptation:** ZAO already has Respect-token gating at the session level (`src/lib/gates/`). Extend this into a **ZAO Feed Rule**: artists can mark posts/tracks as "supporter-only" — visible only to wallets that have sent a tip or hold a minimum Respect score. The gate check is a server-side middleware call in the existing feed API route (`src/app/api/social/`). No new contract needed — reuse the Respect balance check already in `lib/gates/checkGate.ts`. The "edit/delete own content" pattern from Grove is also worth adopting: ZAO casts are currently Farcaster-immutable, but Supabase-cached posts could surface an edit history UI.

**Related research:** `research/280-fid-registration-x402-deep-dive/` covers on-chain identity and access patterns.

## 3. Notion — Row-Level Database Permissions (2025–2026)

**What they did:** Notion added row-level access controls to databases: each collaborator sees only rows where they are assigned via a Person property. A "Can Create Pages" toggle (March 2026) lets contributors add new entries without full edit rights. Result: one shared database serves as a multi-artist pipeline where each artist only sees their own submissions, not other artists' unreleased work.

**Pillar:** Artist Development + Governance — structured workflows that respect privacy between artists on the same label.

**ZAO adaptation:** ZAO's governance system (`src/app/(auth)/governance/`) and song submission flow (`src/components/music/SongSubmit.tsx`) currently treat all members as a flat pool. Apply the Notion pattern to a **ZAO Release Pipeline**: a Supabase table (`release_pipeline`) with Supabase RLS policies scoped to `auth.uid()` matching the submitting artist's FID mapping. Artists see only their own tracks in the pipeline view; admins see all. The existing RLS infrastructure is already in place — this is a policy addition + a new `ReleasePipeline.tsx` component that queries with the scoped client. Governance proposals already use Respect-weighted visibility; this extends that principle to artist-private pre-release data.

**Related research:** `research/music/` covers music curation and submission patterns.
