**North star:** How does ZAO OS become the operating system for a decentralized music label?
**Pillars:** (1) Artist Development · (2) Distribution · (3) Revenue · (4) Governance · (5) Culture

## 1. Apple Music — Playlist Playground (March 2026)

**What it does:** iOS 26.4 shipped Playlist Playground — Apple Music's first generative AI feature. Users describe what they want in natural language ("rainy day hip-hop for studying, no features") and the system builds a playlist, explains its reasoning, and lets users iterate ("make it more lo-fi", "add more 90s"). The key design decision: the AI explains *why* each track was chosen, building trust rather than hiding the algorithm. Paired with a new fullscreen album/playlist view redesign and live event listings embedded directly on artist profiles (via Bandsintown integration — artists toggle one setting to push concert dates into their Apple Music page).

**Pillar served:** Culture + Artist Development. The artist-profile event integration is the sleeper feature — it collapses discovery, social proof, and ticketing into one surface without requiring artists to do extra work.

**ZAO adaptation:** ZAO Radio already has a respect-weighted curation engine (`src/lib/music/curationWeight.ts`). Add a **natural-language radio prompt** to the top of the radio view: "Tell ZAO what you want to hear right now." Route through the existing queue system, but filter + reorder using a simple Claude API call against the current track library. No new infrastructure — just a new input + an API route at `src/app/api/music/radio-prompt/route.ts` that calls the existing curation weight logic with NL-derived filter params. The "why" transparency layer (explain which artists + tags matched) maps directly to ZAO's build-in-public ethos. Target: `src/components/music/RadioView.tsx`, `src/hooks/useRadio.ts`.

**Related research:** `research/050-the-zao-complete-guide/` (radio curation), Apr 08 entry (Notion Agents natural-language interaction model is the same underlying pattern).

## 2. Splice — Artist-Centric AI + Spitfire Instruments (Dec 2025 – Apr 2026)

**What it does:** Two moves that work together: (1) Splice acquired Spitfire Audio (April 2026, ~$50M) — giving it 1,200+ orchestral/cinematic presets playable directly in DAW plugins without leaving Splice. (2) A UMG partnership (December 2025) lets UMG artists bring *their own sounds* into Splice's AI workflows — so a signed artist's vocal chop or drum break can become a licensable sample that other creators pay to use, with the originating artist earning royalties. The key design decision: the originating artist controls what enters the AI training set and sets the price. Creative control is the product, not an afterthought.

**Pillar served:** Revenue + Artist Development. The sample-ownership model is a new revenue stream that doesn't require streaming scale — a single distinctive sound can earn passively across thousands of other producers' projects.

**ZAO adaptation:** ZAO members already submit tracks via `SongSubmit`. Extend this with a **Sample Vault**: members can upload stems/samples with a Respect-weighted price floor. Other members (or external buyers via a public route) pay in USDC; revenue splits to the submitter + a ZAO treasury cut (configured in `community.config.ts`). This is Artist Development (teaching members to think in IP, not just releases) and Revenue (passive income layer). No blockchain needed initially — store samples in Supabase Storage, track purchases in a `sample_purchases` table, pay out via existing treasury logic. Target: `src/app/api/music/samples/route.ts` (new), `src/components/music/SampleVault.tsx` (new), extend `src/components/music/SongSubmit.tsx` to include stem upload option.

**Related research:** `research/051-zao-whitepaper-2026/` (Revenue pillar), Apr 02 entry (DistroKid zero-touch distribution — Sample Vault is a parallel passive income layer).

## 3. Metalabel — Artist Corporations + Shared Treasury (2025)

**What it does:** Metalabel launched A-Corps (Artist Corporations) in 2025 — a group structure where multiple collaborators form a named collective, publish releases under a shared identity, and split revenue automatically from a shared treasury. Each A-Corp has its own release page, collector base, and financial history. The UX is deliberately minimal: create a name, invite members, set split percentages, publish. The "release list + pre-order" feature (launched 2026) lets A-Corps announce upcoming drops, build a collector waitlist, and auto-fulfill when the release goes live — all in one flow with no separate mailing list tool needed.

**Pillar served:** Governance + Distribution + Revenue. The A-Corp model is the closest existing analogue to what ZAO wants to be at the collective level — a named entity that ships creative work, holds money, and distributes upside automatically.

**ZAO adaptation:** ZAO is already a collective with Respect-weighted governance. The gap is **project-level sub-collectives**: when 3 ZAO members want to co-release an EP, there's no ZAO-native way to form a project group, set revenue splits, and publish together. Build **ZAO Projects**: a lightweight sub-collective layer where any member can spin up a project (name + collaborators + split %s), link it to a governance proposal for treasury co-funding, and publish a release page. The release page auto-posts to Farcaster/Bluesky/X via existing publish infrastructure (`src/lib/publish/`). Target: `src/app/(auth)/projects/` (new route group), `src/app/api/projects/route.ts`, `src/components/projects/ProjectCard.tsx`. This is the Distribution pillar's biggest gap — ZAO currently only distributes content *about* music, not music releases themselves.

**Related research:** `research/050-the-zao-complete-guide/` (governance + distribution), Apr 03 entry (Mirror.xyz — same "collective publishes together" pattern via NFT-backed writing). `research/051-zao-whitepaper-2026/` (Distribution pillar).
