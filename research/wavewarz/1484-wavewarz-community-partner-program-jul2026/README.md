# 1484 — WaveWarZ Community Partner Program (July 2026)

> **Status: AGREED — awaiting Solana bridge test before public launch.** Program mechanics and eligibility formally documented here. Do not promote until the Base↔SOL bridge test passes (doc 950, board task ad0218d3).

## What This Is

A 50/50 profit-split program for community builders who submit ideas *and* implementations for the WaveWarZ protocol. ZAO provides full financial backing; the partner brings the idea and builds it.

**The origin quote** (Zaal, doc 950 lines 1159-1162): *"officially agreed to like community partners can come in with an idea for wave wars... we'll fully back it with a split of 50, 50"*

This is distinct from the existing partner integrations (Coinflow, Juke, Magnetiq — see doc 1343). Those are platform-level partnerships. Community partners are individual builders or small teams extending the WaveWarZ protocol.

---

## Program Mechanics

### The Split

| Bucket | Share |
|--------|-------|
| Community Partner | 50% of net revenue generated by the feature/extension |
| ZAO Treasury | 50% |

"Net revenue" = gross fees from the partner-built feature after Solana transaction costs.

### What Counts as a Qualifying Submission

A qualifying submission has two parts — both required:

1. **Idea**: A clearly scoped extension of WaveWarZ protocol functionality. The idea must be new (not already in the WaveWarZ roadmap) and must generate measurable revenue or SOL volume for the platform.
2. **Implementation**: Working code that Zaal can review and deploy. No speculative proposals — ship first.

### Eligible Extensions (Examples)

| Example | Revenue Mechanism |
|---------|-------------------|
| Genre-specific battle rooms | Custom battle creation fee |
| Artist rivalry leaderboard with prediction tracking | Subscription or sponsored |
| Beat-battle format variant (instrumental only) | Per-battle fee variant |
| Fan-page bet integration (audiences bet on their favorite artist) | Expanded trader pool → more volume |
| Clipping agent with built-in WW referral loop | SOL volume attributed to referred registrations |
| ZAOstock ticket gating via WaveWarZ battle history | Protocol fee from gate queries |

### What Is NOT Eligible

- Building something the core team already planned (check doc 1341 roadmap first)
- Front-end reskins with no new protocol function
- Ideas without working code at submission
- Anything requiring Zaal's private keys or admin access at runtime

## Submission Process

1. **Build it.** Working prototype that Zaal can test on devnet.
2. **Open a GitHub issue** in the WaveWarZ repo with: (a) problem statement, (b) what you built, (c) how revenue is attributed to your feature.
3. **Zaal reviews within 5 business days.** Accept = merge + 50/50 split agreement in writing (ZAOOS doc). Decline = specific feedback.
4. **Deploy to mainnet.** Revenue split activated on mainnet deployment.
5. **ZAO pays out monthly** from the protocol fee stream (doc 1321) — partner's 50% converted to USDC or SOL per their preference.

## Key Decisions / Recommendations

| # | Decision |
|---|----------|
| 1 | **GATE: Do not publicly announce until Solana bridge test passes.** The bridge is the payment rail for cross-chain partners. Announcing before it works sets a promise ZAO can't keep. |
| 2 | **First partner is Ohnahji** (implicit from doc 950). His WaveWarZ clipping agent work (doc 1293 pattern) is the prototype case. The test run informs whether the payout mechanics work end-to-end. |
| 3 | **50/50 is the canonical split** — not negotiable per Zaal's public statement. Individual deals that deviate create confusion and precedent risk. If a partner wants a higher share, the answer is "the split is 50/50, but your feature determines how big 50% is." |
| 4 | **Revenue attribution must be on-chain or API-verifiable.** No self-reported numbers. Every qualifying feature must emit a trackable event (battle creation, fee event, API call) that ZAO can independently measure. |
| 5 | **ZAOOS doc = the agreement.** Each accepted partner gets their own ZAOOS doc (sub-doc of this one) that records: partner name, feature built, deployment date, attribution method, first payout date. No separate legal contract required at this stage. |

## Why This Matters (Grant / Press Frame)

From the program announcement perspective:
- "WaveWarZ now offers a 50/50 profit split for community builders who extend the platform" = rare in music tech (most platforms extract from creators; WaveWarZ splits with builders too)
